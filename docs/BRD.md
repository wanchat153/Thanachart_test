## 2. วัตถุประสงค์และบริบททางธุรกิจ

ลูกค้าต้องเลือกสินค้าจากรายการที่มีอยู่จริง เห็นราคาและยอดคงเหลือที่เป็นปัจจุบัน แล้วชำระเงินได้โดย **ระบบต้องไม่ขายของเกินจำนวนที่มีอยู่จริง** แม้มีผู้ซื้อหลายคนกดพร้อมกัน

ปัญหาที่ระบบนี้แก้:

1. **ขายเกินสต๊อก (overselling)** — เกิดเมื่อระบบอ่านยอดคงเหลือแล้วค่อยหักในสองจังหวะ สองคำสั่งซื้อที่เข้ามาพร้อมกันจะเห็นยอดเดียวกันแล้วขายของชิ้นเดียวกันซ้ำ
2. **ราคาถูกแก้จากฝั่งลูกค้า** — ถ้าระบบเชื่อราคาที่เบราว์เซอร์ส่งมา ผู้ใช้แก้ราคาเองได้
3. **บิลไม่ตรงกับของที่ตัดจริง** — ถ้าการตัดสต๊อกกับการออกบิลไม่อยู่ในธุรกรรมเดียวกัน อาจตัดของแล้วบิลไม่เกิด หรือกลับกัน

## 5. กฎธุรกิจ (Business Rules)

กฎเหล่านี้บังคับใช้ที่ **ฝั่งเซิร์ฟเวอร์และฐานข้อมูล** ไม่ใช่แค่ซ่อนปุ่มในหน้าเว็บ — แก้ค่าที่เบราว์เซอร์แล้วก็ยังผ่านไปไม่ได้

| รหัส | กฎ | รายละเอียด |
|---|---|---|
| **BR-01** | ราคาและยอดคงเหลืออ่านจากฐานข้อมูลเสมอ | ตอนชำระเงินระบบอ่านราคาจากตาราง `Product` ใหม่ทุกครั้ง ไม่รับราคาที่เบราว์เซอร์ส่งมา ถ้าผู้ใช้แก้ราคาในเครื่องตัวเองจะไม่มีผลใด ๆ |
| **BR-02** | ตะกร้าเก็บแค่รหัสสินค้ากับจำนวน | ข้อมูลตะกร้าอยู่ในเบราว์เซอร์ และ**จงใจไม่เก็บราคา** เพราะราคาอาจเปลี่ยนหลังผู้ใช้ทิ้งตะกร้าไว้ข้ามวัน |
| **BR-03** | เปิดหน้าเมื่อไร ระบบปรับตะกร้าให้ตรงสต๊อกล่าสุดทันที | ถ้าสต๊อกลดลงระหว่างที่ทิ้งตะกร้าไว้ ระบบตัดยอดส่วนเกินออกเองและแจ้งว่ารายการไหนถูกปรับด้วยเหตุผลอะไร (สต๊อกไม่พอ / สินค้าหมด / ไม่มีสินค้าแล้ว) |
| **BR-04** | สั่งซื้อเกินยอดคงเหลือไม่ได้ | ปุ่มเพิ่มจำนวนถูกปิดเมื่อจำนวนในตะกร้าเท่ากับสต๊อกที่มี และสินค้าที่หมดแล้วกดใส่ตะกร้าไม่ได้ |
| **BR-05** | หนึ่งบิลห้ามมีรหัสสินค้าซ้ำ | ส่งรหัสเดียวกันสองรายการในบิลเดียว → ปฏิเสธทั้งบิล (HTTP 400) ต้องรวมเป็นรายการเดียวแล้วเพิ่มจำนวน |
| **BR-06** | จำนวนสั่งซื้อต้องมากกว่า 0 | จำนวนติดลบหรือศูนย์ถูกปฏิเสธก่อนแตะฐานข้อมูล (HTTP 400) |
| **BR-07** | ตัดสต๊อกกับตรวจสอบยอดทำในคำสั่งเดียว | ใช้คำสั่งอัปเดตแบบมีเงื่อนไข "หักได้เฉพาะเมื่อของยังพอ" ทำให้สองคำสั่งซื้อที่เข้ามาพร้อมกันขายของชิ้นเดียวกันซ้ำไม่ได้ |
| **BR-08** | ของไม่พอแม้รายการเดียว = ยกเลิกทั้งบิล | ไม่ขายบางส่วน ระบบย้อนธุรกรรมทั้งหมดแล้วตอบ HTTP 409 พร้อมบอกว่าสินค้าตัวไหนเหลือเท่าไร **ตะกร้าของผู้ใช้ไม่ถูกล้าง** |
| **BR-09** | บิลบันทึกราคา ณ เวลาที่สั่งซื้อ | ราคาต่อหน่วยในบิลเป็นภาพนิ่ง (snapshot) ราคาสินค้าเปลี่ยนภายหลังแล้วบิลเก่าไม่เปลี่ยนตาม |
| **BR-10** | ทุกการตัดสต๊อกถูกบันทึกเป็นประวัติ | เขียนลงตาราง `StockTransaction` พร้อมจำนวนติดลบและเลขที่บิลที่อ้างอิงถึง ตรวจย้อนหลังได้ว่าของหายไปไหน |
| **BR-11** | บิลที่สร้างมีสถานะ `Paid` เสมอ | เฟสนี้ถือว่ากดชำระเงิน = ชำระเสร็จสมบูรณ์ ยังไม่มีสถานะรอชำระ ยกเลิก หรือคืนเงิน |
| **BR-12** | ค้นหาจากชื่อก่อนรหัสเสมอ | พิมพ์ 1 ตัวอักษรก็ค้นทันที · ผลลัพธ์เรียงตามความตรง: ชื่อขึ้นต้น → ต้นคำในชื่อ → กลางชื่อ → รหัสขึ้นต้น → กลางรหัส · อันดับเท่ากันเรียงตามรหัสเพื่อให้ผลคงที่ |

### 5.1 กฎที่ฐานข้อมูลบังคับเป็นด่านสุดท้าย

แม้โค้ดจะพลาด ฐานข้อมูลก็ยังปฏิเสธ:

| Constraint | ความหมายทางธุรกิจ |
|---|---|
| `CK_Stock_Quantity_NonNegative` | สต๊อกติดลบไม่ได้ |
| `CK_Product_UnitPrice_NonNegative` | ราคาสินค้าติดลบไม่ได้ |
| `CK_OrderDetails_Quantity_Positive` | รายการในบิลต้องมีจำนวนมากกว่า 0 |
| `OrderDetails.SubTotal` เป็น computed column `PERSISTED` | ยอดรวมต่อรายการคำนวณโดยฐานข้อมูล แอปเขียนค่าเองไม่ได้ จึงเพี้ยนไม่ได้ |

## 7. User Flow

```
[ค้นหา (ไม่บังคับ)]           [เลือกสินค้า]              [ตรวจตะกร้า]            [ชำระเงิน]
พิมพ์ชื่อ/รหัส 1 ตัวอักษร  →  ปรับจำนวนด้วย − / +   →  แก้จำนวน/ลบรายการ   →  กด "ชำระเงิน" 1 ครั้ง
เลือกจากรายการแนะนำ           กด "ใส่ตะกร้า"            ระบบปรับให้ตรงสต๊อก      ระบบตัดสต๊อก+ออกบิล
(กรองเหลือตัวเดียว)           ปุ่ม + ตันที่ยอดคงเหลือ    ล่าสุดพร้อมแจ้งเหตุผล    แสดงใบสรุป + เลขที่บิล
```

**เส้นทางที่ไม่ใช่กรณีปกติ**

- สินค้าหมดตั้งแต่แรก → กดใส่ตะกร้าไม่ได้เลย (ปุ่มถูกปิด)
- สต๊อกลดลงระหว่างทิ้งตะกร้าไว้ → ระบบตัดยอดส่วนเกินตอนเปิดหน้า พร้อมข้อความแจ้ง (BR-03)
- ถูกซื้อตัดหน้าตอนกดชำระเงิน → ทั้งบิลถูกยกเลิก ตะกร้าไม่ถูกล้าง ยอดคงเหลือถูกดึงใหม่ให้เห็นความจริง (BR-08)

## 11. ข้อจำกัดและความเสี่ยงที่รับรู้แล้ว

| # | ข้อจำกัด | ผลกระทบ | ทางแก้ในอนาคต |
|---|---|---|---|
| 1 | **ไม่มีการยืนยันตัวตน** — ทุก endpoint เปิดสาธารณะ | ใครก็ยิง `POST /api/checkout` ตัดสต๊อกได้ | เพิ่ม authentication ก่อนขึ้น production |
| 2 | ตะกร้าอยู่ในเบราว์เซอร์ | ล้างข้อมูลเบราว์เซอร์ = ตะกร้าหาย · ใช้ข้ามเครื่องไม่ได้ | ผูกตะกร้ากับบัญชีผู้ใช้ |
| 3 | ไม่มีประวัติคำสั่งซื้อฝั่งลูกค้า | ลูกค้าดูบิลย้อนหลังไม่ได้ (ข้อมูลอยู่ใน DB แล้ว) | เพิ่มหน้า `GET /api/orders` |
| 4 | บิลไม่มีสถานะอื่นนอกจาก `Paid` | ยกเลิก/คืนเงินทำไม่ได้ | เพิ่ม state machine ของบิล |
| 5 | ไม่มี CI | ไม่มีอะไรบังคับให้รันเทสต์ก่อน merge | ตั้ง pipeline รัน `dotnet test` + `npm test` ทุก PR |
| 6 | สินค้าไม่มีรูปภาพในฐานข้อมูล | หน้าเว็บใช้ภาพพื้นหลังที่สร้างจากรหัสสินค้าแทน | เพิ่มคอลัมน์รูปภาพ + ที่เก็บไฟล์ |
